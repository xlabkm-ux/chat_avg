# 🤖 AGENT: Инструкции для AI-агента Antigravity (ChatAVG v2.3)

Этот файл содержит обязательные правила поведения агента в данном рабочем пространстве. Он объединяет наработанный ценный опыт (строгая итеративность, самотестирование) с новыми архитектурными стандартами ChatAVG Agent Platform v2.3.

---

## 🏗️ 1. Архитектурная дисциплина (Для v2.3)

**ChatAVG v2.3 — это Meaning-first Agent Execution Platform, а не просто чат.**
Агент обязан соблюдать архитектурные принципы, описанные в `docs/02_architecture/ARCHITECTURE_OVERVIEW_V2_3.md`:
1. **Fast path first:** Простой диалог не использует песочницы (sandbox) или сложный поиск (RAG).
2. **Gateways:** Инференс нейросетей идет через Model Gateway (LiteLLM), внешние инструменты — через MCP Tool Gateway. Не смешивать их.
3. **Durable Runtime:** Долгие миссии выполняются через Temporal (или его аналог), они не должны висеть в памяти HTTP-запроса.
4. **Изменение архитектуры:** Любые фундаментальные архитектурные решения фиксируются в `docs/03_adr/ADR_INDEX.md` (Architecture Decision Records).

## 📋 2. Погружение в контекст (В начале диалога)

**В начале КАЖДОГО нового чата агент ОБЯЗАН:**
1. Прочитать актуальный бэклог проекта: `./PROJECT_BACKLOG.md`.
2. Прочитать карту архитектуры проекта: `./PROJECT_MAP.md`.
3. Ознакомиться с `UI_AUDIT.md` (при работе с фронтендом).
4. Заглянуть в папку `docs/`, чтобы понять текущую спецификацию изменяемого модуля.
5. Кратко сообщить пользователю: текущий статус (какой этап в работе, что будем делать).
6. **Не требовать от пользователя повторного объяснения контекста** — агент восстанавливает его самостоятельно.
7. **Перед правкой любого файла** — проверить в `PROJECT_MAP.md` зависимости этого файла и убедиться, что правки не сломают систему.

## ⚠️ 3. Правило планирования (ОБЯЗАТЕЛЬНО)

**Прежде чем писать код для новых функций или рефакторинга, агент ОБЯЗАН:**
1. Составить краткий технический план предлагаемых изменений.
2. Сообщить этот план пользователю.
3. **Дождаться явного утверждения** плана.
4. Приступать к выполнению только после получения подтверждения.

## 🔄 4. Итеративный процесс разработки (Спринты)

**Агент ОБЯЗАН придерживаться строгого цикла:**
1. **Дробление**: Реализовывать этап короткими итерациями, избегая масштабных правок за один раз.
2. **Завершение спринта**: Каждая итерация должна заканчиваться цепочкой действий:
    - **Тестирование**: Проверка функционала и отсутствия регрессий.
    - **Обновление карты проекта**: Запустить `node dev_studio/refresh.js` перед фиксацией.
    - **Коммит**: Фиксация изменений в Git с информативным сообщением.
    - **Пуш**: Отправка кода в удаленный репозиторий.

## ✅ 5. Чеклист выполненных работ

**После завершения каждого этапа агент ОБЯЗАН:**
1. Обновить чеклист в файле `PROJECT_BACKLOG.md`.
2. Отметить выполненные пункты символом `✅` и указать дату.
3. Записать краткий итог этапа: что сделано, какие файлы затронуты.
4. **Не переходить к следующему этапу** без обновления чеклиста.

*Пример формата:*
```markdown
- [x] ✅ Этап 1 выполнен — 2026-05-07
      Файлы: src/gateway/model.js
      Итог: Перевели вызовы OpenAI на LiteLLM прокси.
```

## 🧪 6. Обязательное самотестирование

**Перед тем как сообщить пользователю о готовности, агент ОБЯЗАН:**
1. Запустить приложение и проверить результат через инструмент `browser_subagent` (для UI).
2. Для Backend/API — проверить работоспособность через вызовы команд curl или тесты.
3. Убедиться, что нет новых ошибок в консоли/логах и старый функционал не сломан.
4. Зафиксировать результат проверки.
5. **Только после успешной самопроверки** сообщить пользователю: «Этап завершён и проверен». Ошибки исправляются самостоятельно до отчета.
6. **Коммит**: Фиксация изменений в Git с информативным сообщением.  **Пуш**: Отправка кода в удаленный репозиторий.

## 🚫 7. Ограничения и запреты

1. **Запрет на Python:** ЗАПРЕЩАЕТСЯ писать серверные компоненты на Python без явного разрешения (исключение — ML-микросервисы, если они зафиксированы в ADR). Основной стек разработки — **Node.js / TypeScript**.
2. **Запрет на старые концепции:** Категорически запрещено возвращаться к отвергнутым идеям (например, MCP для инференса моделей или использование SQLite как полноценного workflow engine). Если есть сомнения, проверьте `docs/02_architecture/ARCHITECTURE_OVERVIEW_V2_3.md`.
