# ChatAVG v2.0 — Приложение A к концепции проекта
## Канонические пользовательские workflow

**Версия:** 1.0  
**Дата:** 9 мая 2026  
**Статус:** Draft / Official Project Appendix / Таблично отформатированная редакция  
**Связанный документ:** `ChatAVG_v2_02_Project_Concept` / «Концепция проекта ChatAVG v2.0»  
**Назначение:** зафиксировать пользовательские workflow, которые проект ChatAVG должен поддерживать в соответствии с концепцией Mission Room, Adequacy Engine, Artifact Workspace и ЭР-подходом; дать каждому workflow краткое русское название для использования в документации, backlog, UI и QA-матрицах.

---

## 1. Назначение приложения

Настоящее приложение уточняет, какие пользовательские workflow являются обязательными для проектирования, реализации, тестирования и приоритизации ChatAVG.

Документ не описывает внутреннюю архитектуру в деталях. Его задача — зафиксировать продуктовую логику: что пользователь должен уметь делать в системе, какие стадии должна проходить работа и какой результат считается корректным.

Главное положение: ChatAVG не должен проектироваться как обычный чат, где пользователь задаёт вопрос, а модель отвечает сообщением. Базовая единица работы ChatAVG — **миссия**, а базовый результат — **артефакт со следом различения, границ, решений и действия**.

---

## 2. Базовая формула пользовательской работы

Канонический пользовательский путь ChatAVG:

```text
ситуация / замысел / конфликт / материал
→ Mission Brief
→ карта различений
→ область определения
→ гипотезы искажений
→ развилки
→ рабочий артефакт
→ человеческое решение
→ действие / проверка / экспорт / Forge
```

Сокращённая формула:

```text
не prompt → answer,
а ситуация → различение → артефакт → выбор → действие.
```

## 3. Принципы workflow-дизайна

Все пользовательские workflow ChatAVG должны соответствовать следующим принципам.

### 3.1. Mission-first

Пользователь приносит не только prompt, а узел реальности: ситуацию, документ, задачу, конфликт, замысел, решение, проект или жизненную развилку.

Система должна уметь распознавать, когда запрос можно обработать как быстрый чат, а когда требуется полноценная миссия.

### 3.2. Artifact-first

Цель сложной работы — не длинный ответ модели, а рабочий артефакт: документ, план, карта решения, концепт, стратегия, код, отчёт, письмо, архитектура или другой результат, который можно использовать.

### 3.3. Domain-boundary-first

Каждое сильное утверждение должно иметь область определения: где оно применимо, на каком уровне работает, чем подтверждено и с какой силой может быть произнесено.

### 3.4. Human sovereignty

Система может различать, структурировать, предлагать варианты, показывать риски и последствия. Но ценностные, траекторные и ответственные решения остаются за человеком.

### 3.5. No hidden authority

ChatAVG не должен создавать впечатление, что знает пользователя глубже него самого, диагностирует его сущность, говорит от имени Глубины или выдаёт окончательную правду.

### 3.6. Risk-based execution

Sandbox, tools, approvals, внешние действия и Forge должны включаться по риску и необходимости, а не по умолчанию для каждого чата.

### 3.7. Fast path preservation

Простые запросы должны оставаться быстрыми. Они не должны оплачивать стоимость полного Mission/Lab/Forge pipeline.

## 5. Канонические пользовательские workflow

### 5.1. Fast Chat Workflow — Быстрый чат

**Краткое русское название:** Быстрый чат

**Назначение:** быстрые ответы, короткие объяснения, локальная правка текста, простой вопрос без сложного контекста.

**Когда используется:**

- пользователь просит краткое объяснение;
- нужна локальная редактура;
- не требуется работа с документами, развилками, role passes или внешними действиями;
- риск ошибки низкий;
- нет необходимости строить Mission Brief и Claim Ledger.

**Пользовательский поток:**

1. Пользователь отправляет простой запрос.
2. Система определяет, что полноценная миссия не требуется.
3. Ответ формируется по fast path.
4. Если данных недостаточно, система явно понижает силу утверждений.
5. При необходимости система предлагает перевести задачу в Mission / Studio / Lab.

**Ожидаемый результат:** быстрый ответ без тяжёлой оркестрации.

**Запрещено:**

- запускать sandbox без необходимости;
- делать heavy RAG по умолчанию;
- выполнять live tool discovery на каждый простой запрос;
- имитировать глубокий анализ там, где был простой вопрос.

---

### 5.2. Mission Room Workflow — Комната миссии

**Краткое русское название:** Комната миссии

**Назначение:** основной режим работы ChatAVG с ситуацией, замыслом, задачей, конфликтом или материалом пользователя.

**Когда используется:**

- пользователь приносит сложную ситуацию;
- требуется не ответ, а проработка;
- есть документы, ограничения, критерии, участники, конфликт или неопределённость;
- нужно получить артефакт, а не сообщение.

**Пользовательский поток:**

1. Пользователь создаёт или продолжает Mission.
2. Система собирает Mission Brief:
   - цель;
   - контекст;
   - критерии успеха;
   - ограничения;
   - исходные материалы;
   - неизвестные;
   - чувствительность материала.
3. Система выделяет данное: факты, события, документы, участники, сроки, ограничения.
4. Система разделяет факты, переживания, интерпретации, гипотезы, оценки и выборы.
5. Система формирует карту различений и область определения ключевых утверждений.
6. Система выявляет возможные искажения.
7. Система создаёт или обновляет рабочий артефакт.
8. Система показывает развилки, если требуется человеческое решение.
9. Пользователь принимает решение, уточняет материал или запускает следующий шаг.

**Ожидаемый результат:** Mission Brief, карта различений, рабочий артефакт, список открытых вопросов и решений.

**Критерий готовности:** пользователь понимает, что дано, что предполагается, где границы знания, какие развилки открыты и какой артефакт формируется.

---

### 5.3. Studio Workflow — Студия

**Краткое русское название:** Студия

**Назначение:** создание, анализ и улучшение рабочих артефактов.

**Типовые задачи:**

- концепция продукта;
- архитектурный документ;
- стратегия;
- roadmap;
- PRD;
- письмо;
- отчёт;
- исследовательская записка;
- план реализации;
- юридический или комплаенс-документ с осторожными формулировками;
- текст, требующий смысловой точности.

**Пользовательский поток:**

1. Пользователь загружает или создаёт материал.
2. Система определяет целевой артефакт.
3. Система строит Claim Ledger для ключевых утверждений.
4. Система проверяет область определения и силу утверждений.
5. Система запускает ограниченные role passes.
6. Роли предлагают Artifact Patches, а не независимые длинные ответы.
7. Пользователь принимает, отклоняет или редактирует изменения.
8. Система собирает новую версию артефакта.

**Ожидаемый результат:** улучшенный артефакт с трассируемыми изменениями.

**Критерий готовности:** каждая значимая правка имеет основание: claim, boundary, риск, источник, конфликт или пользовательское решение.

---

### 5.4. Lab Workflow — Лаборатория

**Краткое русское название:** Лаборатория

**Назначение:** глубокая работа со сложными человеческими, командными, системными и смысловыми задачами.

**Когда используется:**

- есть конфликт целей или ценностей;
- цена ошибки высока;
- пользовательский материал чувствителен;
- требуется многоуровневое различение;
- есть риск самообмана, языковой подмены, ролевой захваченности или системной слепоты;
- нужен human-in-the-loop.

**Пользовательский поток:**

1. Система определяет, что задача требует Lab-режима.
2. Пользователь видит, почему включается более глубокий режим.
3. Система строит расширенный Mission Brief.
4. Система выполняет полный протокол:
   - Observation Map;
   - Claim Ledger;
   - Domain Boundary Map;
   - Language Risk Map;
   - Distortion Hypotheses;
   - Conflict Cards;
   - Trajectory Questions;
   - Artifact Guidance.
5. Система показывает прогресс по стадиям.
6. Пользователь принимает решения в развилках.
7. Система собирает артефакт и план проверки действием.

**Ожидаемый результат:** не окончательный вердикт, а рабочая карта ситуации, развилки, осторожные гипотезы и артефакт дальнейшего действия.

**Запрещено:**

- диагностировать пользователя;
- утверждать сущностный запрос;
- скорить Глубину;
- выдавать сильное переживание за знание о реальности;
- подменять человеческий выбор рекомендацией системы.

---

### 5.5. Adequacy Mapping Workflow — Карта адекватности

**Краткое русское название:** Карта адекватности

**Назначение:** отдельный workflow для проверки материала на искажения, границы и силу утверждений.

**Основные выходы:**

| Выход                 | Назначение                                                  |
| --------------------- | ----------------------------------------------------------- |
| Observation Map       | Что реально предъявлено в материале.                        |
| Claim Ledger          | Утверждения с типом, уровнем, основанием, силой и границей. |
| Domain Boundary Map   | Где выводы имеют право работать.                            |
| Language Risk Map     | Где слово или термин создаёт иллюзию понимания.             |
| Distortion Hypotheses | Осторожные гипотезы возможных искажений.                    |
| Conflict Cards        | Развилки, требующие выбора или проверки.                    |
| Artifact Guidance     | Что изменить, чтобы артефакт меньше искажал ситуацию.       |
| Trajectory Questions  | Вопросы о направлении, повторяемости и цене выбора.         |

**Пользовательский поток:**

1. Пользователь просит проверить текст, решение, ситуацию или артефакт.
2. Система выделяет claims.
3. Система классифицирует claims:
   - fact;
   - experience;
   - interpretation;
   - hypothesis;
   - metaphor;
   - risk;
   - recommendation;
   - value_choice.
4. Система определяет reality level:
   - material;
   - psychic;
   - social;
   - linguistic;
   - systemic;
   - trajectory;
   - indirect_depth;
   - unknown.
5. Система задаёт допустимую силу вывода.
6. Система показывает пользователю наиболее важные искажения и границы.

**Ожидаемый результат:** пользователь видит, где текст или решение говорит сильнее, чем имеет право, и что нужно уточнить.

---

### 5.6. Conflict Card Workflow — Карта развилки

**Краткое русское название:** Карта развилки

**Назначение:** обработка развилок, где система не должна принимать решение за человека.

**Когда создаётся Conflict Card:**

- конфликт целей;
- конфликт ценностей;
- конфликт ресурсов;
- конфликт между фактом и интерпретацией;
- конфликт между быстрым решением и траекторной ценой;
- риск, что техническое решение скрывает человеческий выбор;
- ситуация, где рекомендация без выбора пользователя будет скрытой подменой ответственности.

**Структура Conflict Card:**

```text
ConflictCard
- title
- situation_summary
- conflicting_options
- evidence
- domain_boundaries
- cost_of_each_option
- risks
- what_system_can_recommend
- what_only_user_can_decide
- required_user_decision
```

**Пользовательский поток:**

1. Система обнаруживает развилку.
2. Система формулирует Conflict Card.
3. Пользователь видит варианты и цену.
4. Пользователь выбирает, откладывает, уточняет или просит альтернативу.
5. Решение фиксируется как DecisionRecord.
6. Артефакт обновляется с учётом решения.

**Ожидаемый результат:** ясное человеческое решение вместо скрытого системного давления.

---

### 5.7. Artifact Workspace Workflow — Рабочий артефакт

**Краткое русское название:** Рабочий артефакт

**Назначение:** ведение живого результата миссии.

**Пользовательский поток:**

1. Система создаёт начальный артефакт на основе Mission Brief.
2. Role passes предлагают patches.
3. Adequacy Engine связывает patches с claims, boundaries и рисками.
4. Пользователь принимает или отклоняет изменения.
5. Система сохраняет версии.
6. Система готовит экспорт или Forge-действие.

**Типы артефактов:**

- markdown-документ;
- концепция;
- стратегия;
- roadmap;
- архитектура;
- план;
- письмо;
- отчёт;
- таблица решений;
- код;
- исследовательская записка;
- decision log.

**Обязательные свойства:**

- versioning;
- patch history;
- связь с claims;
- связь с решениями пользователя;
- открытые границы;
- список unresolved questions;
- export readiness.

---

### 5.8. Knowledge / RAG Workflow — Работа с источниками

**Краткое русское название:** Работа с источниками

**Назначение:** работа с документами, источниками, базами знаний и внешним контекстом.

**Пользовательский поток:**

1. Пользователь задаёт вопрос по материалам или прикладывает документы.
2. Система определяет, нужен ли retrieval.
3. Система выбирает режим:
   - no_retrieval;
   - fast;
   - balanced;
   - max_quality.
4. Система получает источники.
5. Система связывает claims с источниками.
6. Система показывает, какие утверждения подтверждены, какие являются гипотезами, а где данных недостаточно.
7. Ответ или артефакт содержит citations/provenance.

**Ожидаемый результат:** не просто ответ “по документам”, а проверяемая карта утверждений и источников.

**Запрещено:**

- включать heavy retrieval по умолчанию;
- выдавать отсутствие источника за подтверждение;
- смешивать факт из документа и интерпретацию модели;
- скрывать деградацию качества retrieval.

---

### 5.9. Tool / Approval Workflow — Инструменты и согласование

**Краткое русское название:** Инструменты и согласование

**Назначение:** безопасное выполнение действий через инструменты и внешние системы.

**Когда используется:**

- чтение внешнего источника;
- изменение файла;
- отправка сообщения;
- создание документа;
- запуск кода;
- изменение проекта;
- вызов API;
- публикация;
- действие с риском потери данных, расходов или внешнего эффекта.

**Пользовательский поток:**

1. Система определяет нужный tool.
2. Policy Engine оценивает риск.
3. Система показывает preview/diff, если действие меняет внешний мир.
4. Пользователь approve / reject / edit.
5. Tool выполняется.
6. Результат фиксируется в audit trail.
7. Артефакт или Mission state обновляются.

**Варианты approval:**

- approve once;
- approve for this run;
- edit then approve;
- reject;
- reject with reason;
- require safer alternative.

**Ожидаемый результат:** действие выполняется прозрачно, с контролем риска и следом решения.

---

### 5.10. Forge Workflow — Кузница

**Краткое русское название:** Кузница

**Примечание по названию:** в продуктовой логике Forge означает не просто «кузницу», а режим материализации результата во внешнем мире.

**Назначение:** материализация результата во внешнем мире.

**Типовые задачи Forge:**

- создать файл;
- экспортировать документ;
- сгенерировать код;
- выполнить анализ в sandbox;
- подготовить публикацию;
- собрать пакет артефактов;
- выполнить browser/code/write workflow;
- сформировать delivery-ready output.

**Пользовательский поток:**

1. Пользователь или система предлагает материализовать результат.
2. Система оценивает риск, стоимость и требуемую изоляцию.
3. Система показывает pre-flight summary.
4. Пользователь подтверждает действие, если требуется approval.
5. Forge выполняет действие в безопасном runtime.
6. Система возвращает результат, logs, artifacts и audit trail.

**Ожидаемый результат:** пользователь получает материализованный результат, а не только текстовую рекомендацию.

**Запрещено:** запускать Forge для каждого обычного текстового запроса.

---

### 5.11. Durable AgentRun Workflow — Устойчивый запуск агента

**Краткое русское название:** Устойчивый запуск агента

**Назначение:** поддержка долгоживущих миссий и агентских run.

**Когда используется:**

- задача длится дольше интерактивного ответа;
- есть ожидание approval;
- нужны retry, cancel, resume;
- используются tools, RAG, Forge или role passes;
- требуется устойчивость к сбоям.

**Состояния AgentRun:**

```text
queued
→ running
→ requires_action
→ waiting
→ completed
→ failed
→ cancelled
→ expired
```

**Пользовательский поток:**

1. Пользователь запускает MissionRun / AgentRun.
2. Система показывает статус и прогресс.
3. Пользователь может отменить, уточнить, approve или вернуться позже.
4. Система сохраняет события и состояние.
5. После завершения пользователь получает артефакт, summary и дальнейшие действия.

**Ожидаемый результат:** долгий workflow не теряется, не дублирует side effects и может быть продолжен.

---

### 5.12. ER Learning Workflow — Обучение ЭР-мышлению

**Краткое русское название:** Обучение ЭР-мышлению

**Назначение:** обучение пользователя ЭР-мышлению на собственном материале.

**Пользовательский поток:**

1. Пользователь приносит ситуацию, текст или решение.
2. Система показывает, как различать:
   - факт;
   - переживание;
   - интерпретацию;
   - гипотезу;
   - оценку;
   - выбор.
3. Система объясняет область определения ключевых claims.
4. Система показывает, где язык создаёт иллюзию понимания.
5. Система предлагает более осторожные формулировки.
6. Пользователь видит, как меняется качество артефакта.

**Ожидаемый результат:** пользователь не только получает ответ, но и учится точнее видеть собственный материал.

**Ограничение:** система не должна превращаться в духовного наставника, терапевта или диагноста.

---

### 5.13. Admin / Observability Workflow — Администрирование и контроль

**Краткое русское название:** Администрирование и контроль

**Назначение:** управление качеством, безопасностью, стоимостью и устойчивостью платформы.

**Основные пользователи:** PM, Tech Lead, Backend, QA, Security, SRE, Semantic Lead, Product Owner.

**Наблюдаемые метрики:**

- latency by mode;
- TTFT для fast path;
- cost per mission;
- cost per adequate artifact;
- approval rate;
- rejection rate;
- tool failure rate;
- retry rate;
- semantic downgrade rate;
- domain boundary violation rate;
- citation correctness;
- artifact patch acceptance rate;
- unresolved conflict rate;
- user overreach feedback.

**Ожидаемый результат:** команда видит не только технические метрики, но и качество смысловой работы системы.

## 6. Runtime modes как пользовательские режимы

| Режим  | Пользовательский смысл                         | Техническая стратегия                      | Основной результат           |
| ------ | ---------------------------------------------- | ------------------------------------------ | ---------------------------- |

## 10. UX-требования к отображению workflow

Пользователь не должен видеть внутреннюю агентную сложность. Интерфейс должен показывать понятные стадии:

```text
Briefing
→ Mapping
→ Role Passes
→ Conflict Resolution
→ Artifact Build
→ Approval / Forge
→ Completed
```

Для пользователя важно видеть:

- что сейчас делает система;
- какие данные использованы;
- где есть неопределённость;
- какие утверждения понижены по силе;
- какая развилка требует его решения;
- как меняется артефакт;
- что можно сделать дальше.

Не следует показывать пользователю полный внутренний Claim Ledger или trace по умолчанию. Эти данные должны быть доступны по раскрытию, но не перегружать основной UX.

## 11. Definition of Done для workflow

Workflow считается реализованным, если выполняются следующие условия.

### 11.1. Product DoD

- Пользователь понимает цель workflow.
- Пользователь видит текущую стадию.
- Пользователь получает понятный результат.
- Пользователь может принять или отклонить ключевые решения.
- Workflow не создаёт скрытый авторитет системы.

### 11.2. Semantic DoD

- Ключевые claims имеют тип, силу и область определения.
- Система различает факт, переживание, интерпретацию, гипотезу и выбор.
- Система понижает силу утверждений при недостатке оснований.
- Система не диагностирует пользователя.
- Система не говорит от имени Глубины, Реальности или сущностного запроса.

### 11.3. Technical DoD

- Workflow имеет state machine.
- Events логируются.
- Ошибки имеют понятную деградацию.
- При необходимости есть cancel/retry/resume.
- Рискованные действия проходят policy/approval.
- Стоимость и latency измеряются.

### 11.4. QA DoD

- Есть unit/contract/integration tests.
- Есть semantic eval cases.
- Есть regression на fast path.
- Есть негативные кейсы: overclaiming, missing boundaries, fake certainty, unsafe advice, tool risk.
- Есть UX smoke для desktop и mobile.

## 12. Anti-workflows

Следующие паттерны запрещены как противоречащие концепции проекта.

| Anti-workflow                        | Почему запрещён                                               |
| ------------------------------------ | ------------------------------------------------------------- |
| Prompt → красивый длинный ответ      | Подменяет миссию сообщением.                                  |
| Sandbox на каждый чат                | Ломает latency и cost без смысловой необходимости.            |
| Multi-agent debate by default        | Создаёт шум, token explosion и имитацию коллективного разума. |
| Depth scoring                        | Создаёт ложный авторитет и псевдодиагностику.                 |
| Hidden recommendation as user choice | Подменяет человеческую ответственность.                       |
| RAG always on                        | Увеличивает latency/cost и создаёт ложную полноту.            |
| Tool execution without preview       | Нарушает безопасность и доверие.                              |
| Claim without boundary               | Создаёт уверенность сильнее основания.                        |
| Therapy-like diagnosis               | Нарушает границы проекта.                                     |
| ER terminology as decoration         | Разрушает смысловую точность подхода.                         |

## 14. Итоговая формула

ChatAVG должен поддерживать не один workflow чата, а систему пользовательских workflow, которые переводят материал пользователя из состояния неразличённой ситуации в состояние проверяемого артефакта и осознанного действия.

Финальная формула:

```text
ChatAVG workflow =
Mission
+ Adequacy Mapping
+ Role Discipline
+ Conflict Cards
+ Artifact Workspace
+ Human Decision
+ Safe Materialization
```

Если workflow не уменьшает искажения, не удерживает область определения, не возвращает человеку ответственность выбора и не ведёт к артефакту или действию, он не является каноническим workflow ChatAVG.
