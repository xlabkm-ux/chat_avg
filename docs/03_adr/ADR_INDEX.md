# Журнал архитектурных решений (ADR Index)

В этом документе фиксируются ключевые архитектурные решения (Architecture Decision Records) для версии 2.3. Внесение изменений в архитектуру должно сопровождаться обновлением/добавлением ADR.

| ID | Тема / Решение | Статус | Контекст |
| :--- | :--- | :--- | :--- |
| **ADR-001** | **Temporal как Durable Runtime** | Зафиксировано | Отказ от попыток написать собственный оркестратор долгих процессов (workflow) поверх SQLite. Temporal берет на себя паузы, retries и ожидание approvals от человека. |
| **ADR-002** | **LiteLLM как Model Gateway** | Зафиксировано | Выделение логики маршрутизации, retry и учета стоимости в отдельный шлюз (Proxy). Код ChatAVG больше не зависит напрямую от SDK конкретного провайдера. |
| **ADR-003** | **E2B / Daytona как Hybrid Sandbox (Forge)** | Зафиксировано | Для сложных агентских миссий, требующих выполнения кода, используется E2B. Но песочницы НЕ запускаются по умолчанию для каждого обычного чата (Fast Path). |
| **ADR-004** | **MCP только как Tool Gateway** | Зафиксировано | Протокол Model Context Protocol (MCP) используется исключительно как контракт для инструментов (tools) и коннекторов к данным. Инференс моделей через MCP не производится. |
| **ADR-005** | **Semantic Shift-Left (Adequacy Engine)** | Зафиксировано | Смысловой слой (оценка адекватности, извлечение утверждений) разрабатывается и тестируется в первых спринтах (до RAG и сложных интеграций), так как является основным продуктовым отличием (USP). |

*(Детальные файлы ADR-XXX создаются в этой же папке по мере необходимости).*
