# ADR-003: E2B Hybrid Sandbox (Forge)

**Status:** Accepted

**Last Updated:** 2026-05-19

**Related:** [SPEC-019](../04_specs/SPEC-019-SANDBOX_MANAGER.md), [RUNBOOK-002](../09_runbooks/RUNBOOK-002-SANDBOX_RECOVERY.md), [THREAT_MODEL.md](../07_security/THREAT_MODEL.md)

## Context

ChatAVG agents occasionally need to execute untrusted code, automate browser interactions, or perform write operations that pose significant security risks if executed directly on the host system. These high-risk actions include:

1. **Code execution**: Running Python/JavaScript/Bash scripts generated by LLMs (potential for malicious payloads)
2. **Browser automation**: Headless browsing for web scraping or testing (risk of XSS, cookie theft, CSRF)
3. **File system writes**: Modifying tenant data or generating artifacts (risk of data corruption, disk exhaustion)
4. **Network egress**: Making outbound API calls from agent actions (risk of SSRF, data exfiltration)

Executing these actions without isolation would expose the platform to:
- **Remote Code Execution (RCE)**: Malicious agent output could compromise the server
- **Data breaches**: Unauthorized access to tenant data or environment secrets
- **Resource exhaustion**: Infinite loops or fork bombs could crash the host
- **Supply chain attacks**: Compromised dependencies in agent-generated code

We evaluated several sandboxing approaches to mitigate these risks while maintaining developer experience and cost efficiency.

## Decision

**Selected: E2B as primary sandbox runtime with LocalProcess adapter for development**

E2B provides secure, isolated cloud environments for executing AI-generated code. Each sandbox is a lightweight VM with configurable CPU/memory limits, network policies, and automatic cleanup.

### Architecture:

1. **SandboxManager**: Orchestrates sandbox lifecycle (provision → run → snapshot → terminate → cleanup)
2. **Execution Classes**: Risk-based routing determines if a sandbox is needed:
   - `text`, `retrieval`, `read`: No sandbox (fast path)
   - `write`, `code`, `browser`, `high_risk`: E2B sandbox required
3. **Egress Policy**: Deny-all outbound traffic by default; tenant-specific allowlists and signed URLs for controlled access
4. **Quarantine**: Suspicious artifacts are locked down before extraction to the main system
5. **Feature Flag**: `SANDBOX_FORGE_ENABLED` allows disabling sandboxes for local development (LocalProcess adapter used instead)

### Cost Model:

E2B pricing: **$0.008 per sandbox-minute** (as of 2026-05-19)

Example monthly costs:
- 100 sandboxes/day × 5 min each × 30 days = 15,000 min/month = **$120/month**
- 500 sandboxes/day × 3 min each × 30 days = 45,000 min/month = **$360/month**

Cost optimization strategies:
- Aggressive TTL (5 min default) to prevent idle sandboxes
- Freeze/snapshot for long-running tasks instead of continuous execution
- Cache sandbox images to reduce cold start times

## Consequences

### Positive

1. **Strong isolation**: E2B sandboxes run in separate VMs with hardware-level isolation; no shared memory or filesystem with the host.
2. **Resource limits**: CPU, memory, and disk quotas prevent denial-of-service attacks from runaway code.
3. **Egress control**: Network policies block unauthorized outbound traffic; only allowlisted domains and signed URLs are permitted.
4. **Automatic cleanup**: Sandboxes are destroyed after TTL expires; no residual state or orphaned processes.
5. **Audit trail**: All sandbox operations (command execution, artifact extraction, egress events) are logged to the audit system.
6. **Fast path preservation**: Low-risk actions (text generation, RAG retrieval) bypass sandboxes entirely, avoiding unnecessary latency and cost.

### Negative

1. **Cost at scale**: At $0.008/min, heavy usage (1000+ sandboxes/day) can exceed $1000/month. Requires careful monitoring and budget alerts.
2. **Cold start latency**: Provisioning a new E2B sandbox takes 2-5 seconds; adds latency to first tool execution in a session.
3. **Vendor dependency**: E2B is a commercial service; outage or price increase would require migration to alternative (Daytona, Firecracker).
4. **Limited debugging**: Inspecting running sandboxes is difficult; developers must rely on logs and artifact extraction for troubleshooting.
5. **Complexity overhead**: SandboxManager adds architectural complexity (state machine, adapter pattern, quarantine logic).

## Alternatives Considered

### 1. Local Process Execution (subprocess.spawn)

**Description**: Execute code directly on the host using Node.js `child_process` with basic resource limits (ulimit, timeout).

**Why Rejected**:
- **Critical security risk**: No true isolation; malicious code can escape via symlink attacks, environment variable injection, or kernel exploits
- **No egress control**: Cannot reliably block outbound network requests from child processes
- **Resource leakage**: Orphaned processes may persist after timeout if SIGKILL fails
- **Production unsuitable**: Only acceptable for local development with trusted inputs
- **Current status**: Used as `LocalAdapter` fallback when `SANDBOX_FORGE_ENABLED=false` (dev-only)

### 2. Docker Containers

**Description**: Spawn ephemeral Docker containers for each code execution with resource limits and network restrictions.

**Why Rejected**:
- **Slower than E2B**: Container startup takes 5-10 seconds (vs. 2-5s for E2B)
- **Image management overhead**: Must build, version, and cache container images for different execution environments
- **Security concerns**: Container escape vulnerabilities (e.g., runc CVE-2019-5736) require constant patching
- **Operational burden**: Managing Docker daemon, image registry, and garbage collection adds complexity
- **Cost trade-off**: While cheaper than E2B (no per-minute fee), operational costs (engineering time, infrastructure) offset savings

### 3. Daytona (Open-source Alternative)

**Description**: Daytona is an open-source sandboxing platform similar to E2B, using Firecracker microVMs.

**Why Rejected**:
- **Maturity gap**: Daytona is less mature than E2B; fewer SDKs, limited documentation, smaller community
- **Self-hosting requirement**: Would need to operate our own Daytona cluster (infrastructure cost + maintenance burden)
- **Unclear cost advantage**: Self-hosting saves per-minute fees but incurs AWS/GCP compute costs + engineering overhead
- **Risk of abandonment**: Open-source project with uncertain long-term viability vs. E2B's venture-backed company

### 4. WebAssembly (Wasm) Sandbox

**Description**: Execute code in Wasm runtimes (Wasmtime, Wasmer) with capability-based security.

**Why Rejected**:
- **Language limitations**: Only supports languages that compile to Wasm (Rust, C, Go); Python/JavaScript support is experimental
- **No browser automation**: Cannot run Puppeteer/Playwright for headless browsing tasks
- **Immature ecosystem**: Limited libraries and tooling for agent workloads
- **Performance overhead**: Wasm translation layer adds latency for compute-intensive tasks

## Validation

Success criteria for E2B sandbox adoption:

1. **Isolation Test**: Execute malicious code (fork bomb, disk fill, network scan) in sandbox; verify no impact on host system.
2. **Egress Denial Test**: Attempt outbound HTTP request to non-allowlisted domain; verify request is blocked and audit event emitted.
3. **TTL Enforcement Test**: Let sandbox idle for >5 minutes; verify automatic termination and resource release.
4. **Quarantine Test**: Generate artifact matching malware signature; verify artifact is locked and security alert triggered.
5. **Cost Monitoring Test**: Track sandbox-minute consumption; verify accurate billing data in CostService.
6. **Fail-Closed Test**: Simulate E2B API failure; verify sandbox calls return error (not silently fall back to LocalAdapter in production).

Current status (Sprint 15): SandboxManager skeleton implemented with E2B adapter interface. LocalAdapter used for development. Production E2B integration pending API key configuration and egress policy implementation.
