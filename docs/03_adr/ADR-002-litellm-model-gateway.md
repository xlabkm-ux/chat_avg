# ADR-002: LiteLLM как Model Gateway

## Статус
Зафиксировано (Sprint 4)

## Контекст (Context)
В рамках развития платформы ChatAVG v2.3 возникла потребность в интеграции с множеством LLM-провайдеров (OpenAI, DeepSeek, Anthropic, Qwen, Gemini, xAI). Прямая реализация адаптеров для каждого провайдера в коде (Backend) привела бы к дублированию логики маршрутизации, обработки ошибок, расчета стоимости токенов и ограничения скорости (rate limiting). Более того, сложно реализовать эффективный механизм отказоустойчивости (fallbacks) на уровне приложения без усложнения бизнес-логики.

## Решение (Decision)
Принято решение вынести логику инференса моделей и маршрутизации в отдельный микросервис — **Model Gateway**, в качестве которого используется **LiteLLM Proxy**.
- ChatAVG Backend взаимодействует с LiteLLM Proxy через стандартный, унифицированный адаптер (`openai_compat`), так как LiteLLM реализует OpenAI-совместимый API.
- LiteLLM берет на себя ответственность за перенаправление запросов на конкретные API (OpenAI, DeepSeek, Google и т.д.), управление ключами доступа (внутри своего энвайронмента), сбор метрик и учет стоимости (token costs).

## Последствия (Consequences)
**Положительные:**
1. **Упрощение Backend-кода:** Не нужно писать и поддерживать сложные адаптеры для каждого нового провайдера (особенно если их API не совместимы с OpenAI).
2. **Отказоустойчивость (Fallbacks):** LiteLLM позволяет прозрачно перенаправлять запрос на другую модель (например, с `gpt-4o` на `deepseek-chat`), если основной провайдер недоступен, без участия Backend-слоя.
3. **Единый мониторинг и Cost Control:** Централизованный контроль расхода токенов и финансов.

**Отрицательные:**
1. **Дополнительный компонент инфраструктуры:** Усложнение развертывания (требуется запускать отдельный процесс/контейнер для LiteLLM).
2. **Задержка (Latency):** Минимальный дополнительный сетевой overhead (проксирование запроса). Однако, это компенсируется быстрым переключением в случае отказов.
