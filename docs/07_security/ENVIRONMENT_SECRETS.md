# Правила работы с .env (Environment Secrets)

## Обзор
Этот документ описывает правила управления конфигурацией и секретами в ChatAVG v2.3 через файлы `.env`. Конфиденциальная информация **никогда** не должна попадать в систему контроля версий (Git).

## Основные правила

1. **Файл `.env.example`**
   - Является шаблоном и **обязан** храниться в репозитории.
   - Не должен содержать реальных секретов (API ключей, паролей, JWT токенов).
   - Все ключи, необходимые для запуска проекта, должны быть перечислены здесь с пустыми или фиктивными значениями.

2. **Локальный файл `.env`**
   - Используется для локальной разработки.
   - **Строго добавлен в `.gitignore`** и не подлежит коммиту.
   - Разработчик должен создать его самостоятельно, скопировав `.env.example` (`cp .env.example .env`).

3. **Формат и Нейминг**
   - Использовать заглавные буквы и подчеркивания (например, `CHATAVG_SECRET`).
   - Использовать префикс проекта `CHATAVG_` для избежания конфликтов в глобальном окружении.

## Обязательные переменные окружения

Ниже приведен список критически важных переменных, которые необходимо настроить:

- `NODE_ENV`: Режим работы (`development`, `production`, `test`).
- `CHATAVG_PORT`: Порт, на котором запускается шлюз (по умолчанию `8200`).
- `CHATAVG_SECRET`: Строка для подписи JWT токенов (минимум 32 случайных символа). **Никогда не используйте значение по умолчанию в production!**
- `CHATAVG_TOKEN_EXPIRY`: Время жизни токена (например, `7d`).
- `CHATAVG_ADMIN_PASSWORD`: Пароль администратора при первом запуске (если не задан, будет сгенерирован автоматически).
- `CHATAVG_ALLOWED_ORIGINS`: Список разрешенных доменов для CORS (через запятую).

## Интеграция с dotenvx
Если проект использует `dotenvx` для шифрования секретов (`◈ encrypted .env [www.dotenvx.com]`), следуйте официальной документации `dotenvx` для управления ключами дешифрования в CI/CD пайплайнах.

## Ротация ключей
- `CHATAVG_SECRET`: При изменении все существующие сессии пользователей (JWT токены) будут сброшены, потребуется повторная авторизация.
- Пароли БД и API-ключи провайдеров управляются через базу данных или конфигурационные файлы, но базовые сетевые ключи настраиваются именно здесь.
